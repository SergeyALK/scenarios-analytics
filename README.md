Данный проект демонстрирует процесс реализации контроля над списанием затрат производственного предприятия. За концептуальную основу взяты реальные рабочие кейсы, применяющиеся в оперативном планировании и мониторинге финансово-хозяйственной деятельности. На практике данный подход использовался в процедуре бюджетного контроля и при ежемесячном закрытии отчетных периодов. 
Поскольку предприятие выпускает продукцию военного назначения, реальные данные были заменены на синтетические без потери смысловой нагрузки результатов.  
Журналы операций, справочники и реестры переработаны и представлены в формате базы данных «db_full.sqlite (схема также приложена - «dbase_scheme.pgerd»).  Наполнение базы данных включает следующие таблицы:

•	«plan» - журнал договоров с информацией о плановых отгрузках, сроках и номенклатуре;

•	«production» - журнал прихода выпущенной продукции на склад;

•	«shipment» - журнал регистрации отгрузок продукции контрагентам;

•	«tnbd» - технический реестр номенклатуры;

•	«calculations» - реестр утвержденных калькуляций;

•	«contracts» - журнал договоров;

•	«costs_items» - справочник статей затрат;

•	«customers» - журнал контрагентов;

•	«departments» - справочник производственных подразделений внутри предприятия;

•	«orders» - реестр производственных заказов (заказ присваивается на связку договор – номенклатурная группа);

•	«price» - справочник действующих цен;

•	«rri» - нормативно-справочная информация по производственным подразделениям;

•	«fact» - журнал фактических операций на текущую дату – преобразованный дебет 20 счета – факт выпуска в разрезе заказов и производственных подразделений;

•	«wip» - незавершенное производство – остаток 20 счета на текущую дату.

Слепок баз данных, используемых для проекта был составлен на 09.11.2025. В технической части проекта присутствует переменная «completed_month», зафиксированная на значении «10», что означает, что в логику расчета заложено 10 закрытых периодов по бухгалтерии. Однако проект будет корректно отрабатывать при любом заданном значении от 1 до 12. Значение «10» выбрано для лучшей наполненности аналитических отчетов и визуализаций. 

Проект разбит на последовательный набор модулей:
1.	«load» - импорт данных посредством sql-запросов. Данные преобразуются в необходимый для формульной работы вид;
2.	«preprocessing» - предобработка данных, добавление необходимых признаков и формульные вычисления;
3.	«scenario_1» - базовый расчет с использованием утвержденных плановых калькуляций;
4.	«scenario_2» - нормативный расчет, от которого в дальнейшем будут рассчитываться ориентиры по списанию затрат;
5.	«control» - проблемные закрытые заказы, не вписывающиеся в требуемый порог рентабельности. Используются для последующей корректировки себестоимости по изделиям;
6.	«guidelines» - ориентиры по списанию затрат для каждого подразделения. Рассчитанная сумма по каждому открытому заказу и статье затрат для устранения перекосов.

   
Каждый модуль также наполнен комментариями, описывающими концептуальную направленность и выделяющий функциональные блоки кода.
В результате выполнения скрипта «main» на выходе обновляется содержимое папки «results». Это набор аналитических отчетов «management_reporting», визуализации «visualization», перечень заказов для дальнейшей проработки «report» а также ориентиры по списанию для каждого подразделения в папке «guidelines». 
На практике данная работа проводилось в несколько итераций с активным взаимодействием с производственными подразделениями.

Проект отражает первый этап данной работы. Дальнейшая процедура заключалась в последовательной загрузке предварительных данных от подразделений и их корректировки с анализом отклонений текущих ключевых метрик от утвержденных бюджетных показателей.
